# Workspace Instructions

## 沟通方式

向使用者汇报时，使用清楚直白的语言说明做了什么、结果如何。写法是：对一个聪明但没在看程式码的人解释。

实际执行过程，包括思考、规划、写程式、除错和解决问题，保持完整的技术严谨度。这条规范只适用于对使用者的沟通方式。

## 回复风格

- 结论、实际改动、原因、验证结果都要详且严谨。
- 不使用工程汇报腔，例如“落地”“推进”等类似表达。
- 直接、专业、去表演化。
- 回复文字永远使用与对方相同的语系，专有名词维持 English。
- 不使用口语化表达，说重点，简单明了。
- 需要时搭配条列式与表格加强可读性。

## 多选项决策

当方案有多个选项时，列出每个选项的优缺点，并明确指出推荐选项与原因。

## 验证标准

开始任务前先定义完成标准。交付前依此验证，发现问题就修好再测，不把未完成的工作交回给使用者。只有确认完成，或遇到真正需要使用者介入的障碍时，才回报。

## Stata 使用规范

- 写论文正式代码、可重复的主程序、完整复现流程时：以 `.do` 文件为主。优先维护清晰、可重复运行的 `.do` 文件，再用 Stata 批处理运行并检查 `.log`、表格、图和导出文件。
- 调试 `.do`、检查数据、逐步跑回归、定位变量或语法错误、查看中间结果时：优先使用 Stata MCP。使用交互式 session 连续执行 `describe`、`summarize`、`tabulate`、`regress`、`reghdfe` 等命令。
- 长时间批量任务、完整样本大规模估计、正式生成论文表格和图形时：优先使用普通 Stata 批处理方式，例如 `StataMP-64.exe /e do xxx.do`。
- 交互式分析、探索数据、试估计、快速验证一条命令或一段代码时：优先使用 Stata MCP。

原则：Stata MCP 用来提高交互式调试和探索效率；正式研究代码仍应沉淀为 `.do` 文件，保证可复现、可审查、可长期维护。

## MATLAB 使用规范

- 正式项目代码、可复现主程序、长期维护的研究流程时：以 `.m` 文件、函数和 MATLAB test 文件为主。
- 调试函数、检查变量、试一小段代码、确认 toolbox 或快速验证 MATLAB 表达式时：优先使用 MATLAB MCP。
- 长时间仿真、大型 batch、正式生成论文或项目结果时：优先使用普通 MATLAB 批处理方式，例如 `matlab.exe -batch "run('main.m')"`。
- 单元测试和代码质量检查时：优先使用 MATLAB MCP。

原则：MATLAB MCP 用来提高交互式调试、环境检查、测试和代码质量检查效率；正式项目代码仍应沉淀为 `.m` 文件、函数和可重复运行的测试。

## LaTeX / MiKTeX 使用规范

在本工作区需要编译 LaTeX 时，默认使用本机 MiKTeX：

- MiKTeX 安装根目录：`D:\application\miktex`
- MiKTeX 可执行文件目录：`D:\application\miktex\miktex\bin\x64`
- 如果 `pdflatex`、`xelatex`、`lualatex`、`bibtex` 等命令不在 `PATH` 中，不要判断为没有 LaTeX 编译器；应直接使用上述目录中的完整路径调用。
- 默认优先尝试项目已有编译脚本；没有脚本时，可直接调用：

```powershell
& "D:\application\miktex\miktex\bin\x64\pdflatex.exe" -interaction=nonstopmode -halt-on-error main.tex
```
